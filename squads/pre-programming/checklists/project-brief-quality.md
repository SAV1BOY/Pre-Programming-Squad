# Checklist: Qualidade do Project Brief

## Propósito
Garantir que o brief do projeto contém todas as informações fundamentais para iniciar o trabalho de pré-programação com clareza e direção.

## Quando Usar
- Ao receber um novo pedido de projeto ou feature
- Antes de iniciar a fase de discovery
- Quando há dúvida se o escopo está suficientemente definido para prosseguir

---

## Checklist

### Problema
- [ ] O problema de negócio está descrito de forma clara e objetiva
- [ ] Existe evidência ou dado que comprova a existência do problema
- [ ] O impacto do problema está quantificado (receita, usuários, operação)
- [ ] O problema é distinguido da solução proposta (não pula direto para "precisamos de X")
- [ ] O contexto temporal está claro (desde quando existe, urgência)

### Objetivo
- [ ] O objetivo está formulado de maneira mensurável
- [ ] Existe diferença clara entre objetivo primário e secundários
- [ ] O objetivo está alinhado com metas estratégicas da empresa/área
- [ ] O resultado esperado está descrito em termos de valor entregue
- [ ] Existe definição de "o que é sucesso" para este projeto

### Stakeholders
- [ ] Todos os stakeholders relevantes estão identificados
- [ ] O sponsor/dono do projeto está definido
- [ ] As expectativas de cada stakeholder estão documentadas
- [ ] Os decisores finais estão identificados (quem aprova o quê)
- [ ] Possíveis conflitos de interesse entre stakeholders foram mapeados

### Restrições
- [ ] Restrições de prazo estão explícitas e justificadas
- [ ] Restrições orçamentárias estão documentadas
- [ ] Restrições técnicas conhecidas estão listadas (stack, infra, compliance)
- [ ] Restrições regulatórias ou legais foram consideradas
- [ ] Dependências de outros times/projetos estão identificadas

### Critérios de Sucesso
- [ ] KPIs ou métricas de sucesso estão definidos
- [ ] Existe baseline atual para comparação
- [ ] O prazo de avaliação do sucesso está definido
- [ ] Critérios de sucesso são verificáveis e não subjetivos
- [ ] Existe alinhamento entre stakeholders sobre o que é sucesso

---

## Critérios de Aprovação
- **Mínimo**: Todos os itens de Problema, Objetivo e Stakeholders marcados
- **Recomendado**: Todos os itens marcados
- **Bloqueante**: Problema não descrito ou objetivo não mensurável

## Sinais de Alerta (Red Flags)
- Brief descreve solução sem explicar o problema
- Nenhum dado ou evidência sustenta a existência do problema
- Stakeholders não identificados ou sem dono claro
- Prazo definido sem justificativa ("precisa para ontem")
- Objetivo vago tipo "melhorar a experiência do usuário" sem métrica

## Agente Responsável
**Agente de Discovery & Framing** — responsável por validar a qualidade do brief antes de iniciar qualquer análise.
