# Nova Feature — Fase 01: Intake

## Objetivo da Fase

Receber e validar a solicitação de nova feature, garantindo que o problema está bem definido, o stakeholder está identificado e há informação suficiente para iniciar a discovery.

## Agentes Envolvidos

- **Agente de Requisitos** (líder da fase) — Valida completude da solicitação
- **Agente Coordenador** — Registra o projeto e aloca recursos

## Inputs

- Solicitação de feature (PRD, ticket, briefing ou comunicação informal)
- Contexto de negócio (OKR ou iniciativa estratégica que motiva a feature)
- Stakeholder primário identificado
- Prioridade preliminar

## Atividades

1. **Receber e registrar solicitação** — Criar registro formal do projeto com ID único, stakeholder e data de entrada
2. **Validar completude mínima** — Verificar que a solicitação contém: problema a resolver, quem é afetado, por que agora
3. **Classificar tipo e complexidade** — Determinar se é feature nova (greenfield), extensão de feature existente ou feature cross-cutting
4. **Identificar dependências iniciais** — Listar sistemas, times e dados que provavelmente serão afetados
5. **Definir canal de comunicação** — Estabelecer canal com stakeholder para dúvidas e validações durante o processo
6. **Agendar kickoff de discovery** — Se intake aprovado, agendar sessão de discovery com stakeholders relevantes

## Outputs

- Registro do projeto com metadados (ID, stakeholder, prioridade, tipo, data)
- Avaliação de completude da solicitação (aprovada ou devolvida com pedidos de esclarecimento)
- Lista preliminar de dependências e times envolvidos
- Agenda de kickoff de discovery

## Gates de Qualidade

| Gate | Critério | Obrigatório |
|------|----------|-------------|
| Problema definido | Solicitação descreve problema concreto, não solução | Sim |
| Stakeholder identificado | Há pessoa responsável para validações | Sim |
| Contexto de negócio | Existe link com OKR, métrica ou iniciativa | Sim |
| Informação suficiente | Agente de Requisitos confirma que há base para discovery | Sim |
