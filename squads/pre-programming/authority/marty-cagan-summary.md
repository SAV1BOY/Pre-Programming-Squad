## Tese Central

A maioria das organizações constrói software de forma fundamentalmente errada: recebem requisitos de stakeholders, transformam em roadmap de features, e entregam o que foi pedido sem questionar se resolve o problema real. Times de produto verdadeiramente eficazes são **empoderados** para descobrir soluções para problemas (não entregar features), usam **discovery** para validar ideias antes de construí-las, e medem sucesso por **outcomes** (resultados de negócio), não por **outputs** (features entregues).

---

## Conceitos-Chave

### 1. Discovery vs Delivery
- **Discovery**: Descobrir o que construir. Validar que a solução é valiosa (customers want it), viável (we can build it), usável (users can figure it out) e viável para o negócio (it works for the business).
- **Delivery**: Construir o que foi descoberto, com qualidade e confiabilidade.

A pré-programação é a ponte entre discovery e delivery.

### 2. Empowered Teams vs Feature Teams
- **Feature Teams**: Recebem roadmap de features, implementam o que foi pedido. Servem o negócio como "fábrica de código".
- **Empowered Teams**: Recebem problemas para resolver e métricas para mover. Têm autonomia para escolher a solução.

### 3. Riscos de Produto
Cagan identifica 4 riscos que devem ser endereçados antes de construir:
- **Value Risk**: Os clientes vão querer/comprar isso?
- **Usability Risk**: Os usuários conseguem usar?
- **Feasibility Risk**: Conseguimos construir com a tecnologia e tempo disponíveis?
- **Business Viability Risk**: Funciona para o negócio (financeiro, legal, compliance)?

### 4. Opportunity Assessment
Antes de qualquer projeto, responder:
- Que problema estamos resolvendo? (objetivo)
- Para quem? (audiência)
- Como saberemos se tivemos sucesso? (métricas)
- Que abordagem vamos testar? (solução candidata)
- Quão confiantes estamos? (nível de risco)

### 5. Product vs Project
Projetos terminam. Produtos evoluem. Pensar em "projetos" leva a mentalidade de "entregar e esquecer". Pensar em "produto" leva a ownership contínuo.

## Aplicação ao Squad

- **4 riscos como checklist de pré-programação**: Todo projeto avaliado pelo squad deve endereçar os 4 riscos (valor, usabilidade, viabilidade, negócio) antes de avançar para codificação.

- **Opportunity Assessment como prerequisite**: Antes de criar design doc ou ADR, o squad deve ter um Opportunity Assessment respondido. Se não há problema claro, não há projeto.

- **Métricas de outcome, não output**: No project brief, métricas de sucesso devem ser de resultado (conversão, retenção, NPS), não de entrega (features lançadas, story points completados).

- **Discovery informando o handoff**: Resultados de discovery (protótipos testados, feedback de usuários, dados de experimentos) devem ser incluídos no pacote de handoff.

- **Questionar o roadmap**: Se o brief diz "construir feature X", o squad deve perguntar "qual problema a feature X resolve?". Se não há resposta clara, o projeto não está pronto.
