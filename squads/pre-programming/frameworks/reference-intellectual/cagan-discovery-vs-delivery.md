# Cagan: Descobrir Antes de Construir

## Título e Propósito

Framework baseado no trabalho de **Marty Cagan** (*Inspired*, *Empowered*). A tese central: **as melhores equipes de produto fazem discovery antes de delivery** — validando que estão construindo a coisa certa antes de investir em construir certo. O propósito é alinhar o trabalho de pré-programação com o discovery de produto.

## Quando Usar

- No início de qualquer iniciativa de produto antes de comprometer com implementação
- Quando há risco de viabilidade (técnica), usabilidade, valor ou viabilidade de negócio
- Quando a equipe está construindo features que ninguém usa
- Para alinhar produto e engenharia sobre o que investigar antes de construir
- Quando stakeholders pressionam por entregas sem validação de demanda

## Conceitos-Chave

1. **Quatro Riscos**: Toda ideia de produto tem riscos de Valor (as pessoas querem?), Usabilidade (as pessoas conseguem usar?), Viabilidade (conseguimos construir?) e Viabilidade de Negócio (funciona para o negócio?).
2. **Discovery vs. Delivery**: Discovery valida os 4 riscos com mínimo investimento. Delivery constrói a solução validada com qualidade de produção.
3. **Protótipo como Ferramenta de Discovery**: Protótipos de alta e baixa fidelidade para testar hipóteses sem construir o produto.
4. **Equipe Empoderada**: Times que recebem problemas para resolver, não features para construir.
5. **Opportunity Assessment**: Antes de investir em discovery, avalie: o problema vale ser resolvido?

## Processo / Passos

### Passo 1 — Identificar os 4 Riscos
Para cada iniciativa, avalie: há risco de valor? Usabilidade? Viabilidade técnica? Viabilidade de negócio?

### Passo 2 — Discovery para Cada Risco
Para cada risco significativo, defina um experimento de validação:
- Valor: entrevistas, pesquisa, fake door test
- Usabilidade: protótipo + teste com usuários
- Viabilidade: spike técnico, protótipo funcional
- Negócio: análise financeira, validação com stakeholders

### Passo 3 — Validar Antes de Comprometer
Só avance para delivery quando os riscos principais estiverem endereçados.

### Passo 4 — Iterar Rapidamente
Discovery não é fase única — é ciclo contínuo de hipótese → teste → aprendizado.

### Passo 5 — Comunicar Resultados
Compartilhe descobertas com stakeholders. Se o discovery revelou que a ideia não funciona, isso é resultado positivo — economizou meses de desenvolvimento.

## Perguntas de Ativação

- "As pessoas realmente querem isso ou estamos assumindo?"
- "Validamos com usuários reais ou com nossa intuição?"
- "Sabemos que é tecnicamente viável ou estamos apostando?"
- "O modelo de negócio suporta essa feature?"
- "Qual é o experimento mais barato que podemos fazer para validar?"

## Output Esperado

Mapa de riscos por iniciativa, experimentos de validação definidos e executados, resultados documentados, decisão Go/No-Go baseada em evidência.

## Armadilhas Comuns

1. **Delivery sem discovery**: Construir features que ninguém pediu, ninguém quer ou ninguém consegue usar.
2. **Discovery infinito**: Validar sem parar para evitar o risco de construir. Paralisia por análise.
3. **Protótipo como produto**: Confundir protótipo de discovery com produto pronto para produção.
4. **Ignorar resultados negativos**: Discovery mostra que a ideia não funciona, mas construímos mesmo assim.
5. **Discovery apenas por produto**: Engenharia deve participar do discovery, especialmente para risco de viabilidade.
