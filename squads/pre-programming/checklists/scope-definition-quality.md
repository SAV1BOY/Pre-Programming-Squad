# Checklist: Qualidade da Definição de Escopo

## Propósito
Garantir que o escopo do projeto está bem delimitado, com fronteiras claras entre o que será feito e o que não será, priorizações definidas e MVP identificado.

## Quando Usar
- Após formalização dos requisitos
- Antes de iniciar estimativas e planejamento
- Quando houver pressão para "incluir mais uma coisinha"

---

## Checklist

### In-Scope / Out-of-Scope
- [ ] Existe lista explícita do que está dentro do escopo
- [ ] Existe lista explícita do que está fora do escopo
- [ ] Cada item out-of-scope tem justificativa documentada
- [ ] Stakeholders revisaram e concordaram com as fronteiras
- [ ] Itens de escopo ambíguo foram resolvidos (não ficaram em "área cinza")

### MVP (Minimum Viable Product)
- [ ] O MVP está definido como o menor conjunto que entrega valor
- [ ] Existe critério claro para o que é "viável" no MVP
- [ ] O MVP é demonstrável e testável por um usuário real
- [ ] Features do MVP estão diferenciadas de features futuras
- [ ] O MVP pode ser entregue dentro das restrições de prazo e recurso

### Cortes e Trade-offs
- [ ] Decisões de corte estão documentadas com justificativa
- [ ] Trade-offs foram discutidos com stakeholders (não decididos unilateralmente)
- [ ] Impacto de cada corte está avaliado (o que se perde ao não fazer)
- [ ] Existe plano para itens cortados (backlog, próxima versão, descartado)
- [ ] Cortes de escopo não comprometem a viabilidade técnica da solução

### Prioridades
- [ ] Existe método de priorização aplicado (MoSCoW, RICE, Kano ou similar)
- [ ] Prioridades refletem valor de negócio, não preferência pessoal
- [ ] Existe diferenciação entre urgente e importante
- [ ] Dependências técnicas influenciaram a priorização quando necessário
- [ ] A priorização foi validada com product owner ou sponsor

### Proteção de Escopo
- [ ] Existe processo para avaliar pedidos de mudança de escopo
- [ ] Critérios para aceitar/rejeitar mudanças estão definidos
- [ ] Impacto de mudanças de escopo em prazo e custo é avaliado antes de aceitar
- [ ] Há responsável designado para proteger o escopo (gatekeeper)
- [ ] Histórico de mudanças de escopo é registrado

---

## Critérios de Aprovação
- **Mínimo**: In-Scope/Out-of-Scope e MVP completos
- **Recomendado**: Todos os itens marcados
- **Bloqueante**: MVP não definido ou fronteiras de escopo ambíguas

## Sinais de Alerta (Red Flags)
- Lista de out-of-scope vazia (tudo está incluído = escopo infinito)
- MVP que não pode ser entregue nas restrições existentes
- Priorização onde tudo é "Must Have"
- Nenhum trade-off documentado (decisões parecem fáceis demais)
- Escopo que cresce a cada reunião sem processo de controle

## Agente Responsável
**Agente de Requirements & Scope** — responsável por definir e proteger as fronteiras do escopo.
