# Nova Feature — Fase 02: Discovery

## Objetivo da Fase

Investigar profundamente o problema, mapear personas afetadas, documentar cenários de uso e identificar restrições que influenciam a solução.

## Agentes Envolvidos

- **Agente de Requisitos** (líder da fase) — Conduz análise de requisitos e cenários
- **Agente de Escopo** — Começa a identificar fronteiras do que está dentro e fora

## Inputs

- Registro de intake aprovado (Fase 01)
- PRD ou briefing do stakeholder
- Dados de uso do produto atual (analytics, suporte, pesquisas)
- Feedback de usuários (se disponível)

## Atividades

1. **Analisar o problema em profundidade** — Ir além do sintoma para entender a causa raiz do problema que a feature resolve. Quantificar impacto com dados.
2. **Mapear personas afetadas** — Identificar todos os tipos de usuário afetados, seus padrões de uso e expectativas. Usar dados reais, não personas fictícias.
3. **Documentar cenários de uso** — Para cada persona, mapear cenários happy path, sad path e edge cases. Cada cenário com pré-condição, ação e resultado esperado.
4. **Identificar requisitos não-funcionais** — Performance, segurança, acessibilidade, internacionalização, compliance. Cada um com métrica quando aplicável.
5. **Levantar restrições conhecidas** — Limitações técnicas, regulatórias, contratuais ou de prazo que influenciam a solução.
6. **Desambiguar terminologia** — Criar glossário de termos do domínio que podem ter interpretações diferentes entre times.
7. **Validar entendimento com stakeholder** — Sessão de validação para confirmar que o entendimento está correto antes de prosseguir.

## Outputs

- Documento de discovery com: problema quantificado, personas com dados, cenários de uso detalhados
- Lista de requisitos não-funcionais com métricas
- Lista de restrições conhecidas
- Glossário de domínio
- Ata de validação com stakeholder

## Gates de Qualidade

| Gate | Critério | Obrigatório |
|------|----------|-------------|
| Problema quantificado | Impacto descrito com números (usuários afetados, custo, frequência) | Sim |
| Personas com dados | Cada persona tem padrão de uso baseado em dados reais | Sim |
| Cenários documentados | Mínimo de 3 cenários por persona (happy, sad, edge) | Sim |
| Stakeholder validou | Ata de sessão de validação assinada | Sim |
| Ambiguidades resolvidas | Termos ambíguos estão no glossário com definição acordada | Sim |
