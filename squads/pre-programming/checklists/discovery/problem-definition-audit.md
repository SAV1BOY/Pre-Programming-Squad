# Checklist: Auditoria de Definição do Problema

## Propósito
Verificar se o problema está claro, bem delimitado e fundamentado em evidências, antes de prosseguir com qualquer análise de solução.

## Quando Usar
- No início da fase de discovery
- Quando há sensação de que "não sabemos exatamente o que estamos resolvendo"
- Antes de gastar tempo avaliando soluções

---

## Checklist

### Clareza do Problema
- [ ] O problema está descrito em uma ou duas frases claras
- [ ] A descrição do problema não contém solução embutida
- [ ] Qualquer pessoa do time consegue explicar o problema sem consultar documentos
- [ ] O problema é descrito na perspectiva de quem sofre com ele (usuário, negócio, operação)
- [ ] Existe distinção entre o problema real e pedidos de funcionalidade

### Delimitação
- [ ] O escopo do problema está definido (o que é e o que não é parte do problema)
- [ ] Fronteiras temporais estão claras (desde quando existe, quando precisa ser resolvido)
- [ ] Fronteiras organizacionais estão claras (quais áreas/times são afetados)
- [ ] O problema está dimensionado (quantos usuários, qual frequência, qual impacto)
- [ ] Problemas adjacentes foram identificados e conscientemente separados

### Fundamentação
- [ ] Existe pelo menos um dado ou evidência que comprova a existência do problema
- [ ] A fonte dos dados é confiável e verificável
- [ ] O problema foi validado com quem o vivencia (não apenas com quem o reportou)
- [ ] Viés de confirmação foi considerado (procurou-se contra-evidências)
- [ ] O tamanho do problema justifica o investimento em resolvê-lo

### Impacto
- [ ] Impacto atual do problema está quantificado (custo, tempo perdido, satisfação)
- [ ] Impacto futuro caso não seja resolvido está projetado
- [ ] Urgência vs importância está avaliada
- [ ] Quem mais sofre com o problema está identificado
- [ ] Existe alinhamento sobre a prioridade deste problema vs outros

---

## Critérios de Aprovação
- **Mínimo**: Clareza e Fundamentação completos
- **Recomendado**: Todos os itens marcados
- **Bloqueante**: Problema não descrito claramente ou sem nenhuma evidência

## Sinais de Alerta (Red Flags)
- Problema descrito como "precisamos de [solução]"
- Nenhum dado sustenta a existência do problema
- Cada stakeholder descreve um problema diferente
- Problema existe há anos e ninguém sabe quantificar o impacto
- Problema tão amplo que não pode ser resolvido em um projeto

## Agente Responsável
**Agente de Discovery & Framing** — responsável por garantir clareza e fundamentação do problema.
