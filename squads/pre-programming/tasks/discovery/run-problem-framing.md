# Task: Executar o Problem Framing

## Objetivo
Enquadrar o problema de forma clara, estruturada e livre de viés de solução, garantindo que a equipe entenda o "porquê" antes de discutir o "como".

## Input Necessário
- Project Brief aprovado.
- Clarificações de requisitos.
- Acesso a stakeholders e usuários afetados.
- Dados quantitativos do cenário atual (se disponíveis).

## Agentes Envolvidos
- **Agente de Discovery:** Facilita a sessão de problem framing.
- **Stakeholders:** Fornecem perspectiva do negócio.
- **Usuários/Representantes:** Fornecem perspectiva de quem vive o problema.
- **Agente de Domínio:** Contribui com conhecimento técnico do contexto.

## Passos

### 1. Preparação
- Reunir todos os dados disponíveis sobre o problema.
- Preparar perguntas usando framework "5 Porquês" ou "How Might We".
- Convidar participantes com perspectivas diversas.

### 2. Declarar o Problema
- Facilitar a construção de uma declaração de problema no formato:
  - **Quem** é afetado?
  - **O que** está acontecendo (ou não acontecendo)?
  - **Quando** e **onde** o problema ocorre?
  - **Qual** o impacto mensurável?
  - **Por que** isso importa?

### 3. Mapear o Cenário Atual (As-Is)
- Documentar o fluxo atual passo a passo.
- Identificar pontos de fricção, gargalos e desperdícios.
- Quantificar o impacto de cada ponto de dor (tempo, custo, satisfação).

### 4. Identificar Causas-Raiz
- Aplicar técnica dos "5 Porquês" para cada ponto de dor principal.
- Diferenciar sintomas de causas.
- Mapear relações causais entre problemas.

### 5. Definir Fronteiras do Problema
- O que está dentro do problema que vamos resolver?
- O que está fora (problemas adjacentes que não atacaremos agora)?
- Quais são as restrições do contexto?

### 6. Documentar no Template de Discovery
- Preencher o template `discovery-template.md` com todos os achados.
- Registrar hipóteses levantadas e perguntas em aberto.

## Output Esperado
- Declaração de problema clara e validada por stakeholders.
- Mapa do cenário atual (As-Is) com pontos de dor quantificados.
- Análise de causas-raiz documentada.
- Discovery template preenchido.

## Checklist de Validação
- [ ] A declaração do problema não contém referências a soluções.
- [ ] O cenário atual foi mapeado com participação de quem vive o problema.
- [ ] Pontos de dor estão quantificados (tempo, custo, frequência).
- [ ] Causas-raiz foram identificadas (não apenas sintomas).
- [ ] Fronteiras do problema estão definidas.
- [ ] Hipóteses estão documentadas com plano de validação.
- [ ] Stakeholders validaram a declaração do problema.
